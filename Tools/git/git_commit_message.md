# Git Commit Message

working dir(작업중인 로컬 디렉터리)에서 `git add`를 하게되면 변경된 파일의 목록이 `index(stage)`에 추가가 되는데, 이 파일의 목록들을 `HEAD(확정본)`에 반영을 시킬 때 `git commit`을 쓰게 된다. 다시 말해 **`HEAD`에 어떤 변화가 반영이 되었는지 설명하기 위한 글** 이 커밋 메시지이다.

이 커밋 메시지를 규칙에 맞춰 좋게 작성해야 하는 이유는 다음과 같다.
- 팀원과의 소통
- 편리한 과거의 기록 추적

## Commit Option

- `-m`   

커밋 메시지를 작성해준다.   
```sh
git add file
git commit -m "FIX ~~"
```

- `-a or --all`

모든 파일을 자동으로 commit해준다. 단, 그리 추천하지 않는다.

```sh
git commit -a -m "ADD ~~~"
```

- `--amend`

원격 저장소로 푸쉬되지 않은 마지막 커밋 메시지를 다시 작성한다.

```sh
git add .
git commit --amend -m "ADD ~~"
```

## 좋은 커밋 메시지 작성법

1. **커밋 유형 지정**
   - FEAT : 새로운 기능의 추가
   - FIX : 버그 수정
   - DOCS : 문서 수정
   - STYLE : 스타일 관련 기능(코드 포맷팅, 세미콜론 누락, 코드 자체의 변경이 없는 경우)
   - REFACTOR : 코드 리펙토링
   - TEST : 테스트 코드, 리펙토링 테스트 코드 추가
   - CHORE : 빌드 업무 수정, 패키지 매니저 수정(ex .gitignore 수정 같은 경우)
2. **제목과 본문을 빈 행으로 분리**
   - 여러 행으로 구성된 커밋 로그를 `-m` 스위치를 사용해서 입력하기는 어렵다. 해서 적합한 편집기를 사용하여 편집을 진행하는 것이 좋다.
3. **제목 행을 50자로 제한**
   - 강제 제한 아님! 다만 제목은 읽기 쉽고 간결하게 표현하기 위한 경험에 의한 규칙!
4. **제목 행의 첫 글자는 대문자로 시작**
   - readme file modification -> X!
   - Readme file modification -> O!
5. **제목 행 끝에 마침표를 넣지 않음**
6. **제목 행에 명령문 사용**
   - 영어 명령문이랑 똑같이 작용!
   - 방을 치워라 (Clean your room)
   - 문을 닫아라 (Close the door)
7. **본문은 72자마다 끊어 줄을 바꿈**
8. **본문을 사용하여 변경한 내용과 이유 설명(어떻게보다는 무엇과 왜!)**
9. **검토자가 원래 문제가 무엇인지 이해한다고 가정X! 확실하게 설명 추가**
10. **자신의 코드가 직관적으로 바로 파악할 수 있다고 생각하지 않기**
11. **팀에서 정한 Commit 규칙 따르기!**
